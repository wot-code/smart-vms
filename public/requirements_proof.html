Smart Visitor Management System — Appendix Document

Proof of Functional Requirements

This document maps the theoretical requirements from section 4.2.1 directly to the concrete system architecture and codebase.

1. Secure Authentication & Role-Based Access

"secure authentication mechanism that allows different categories of users... based on their roles."
  • Proof of Implementation: The system utilizes Laravel Breeze for secure, cryptographically hashed passwords. Role-Based Access Control (RBAC) is achieved through a custom role column on the users table. The application uses server-side middleware to restrict routes (e.g., a Guard returning a 403 Forbidden error if attempting to access Admin analytics).

2. Digital Visitor Registration

"registration of visitors by capturing essential details such as name, contact information, purpose of visit..."
  • Proof of Implementation: Handled entirely by the VisitorRegistrationForm.php Livewire component. The system captures standard text data and utilizes an HTML5 Canvas to capture encrypted Base64 digital signatures, eliminating paper-based loopholes.

3. Unique QR Code Generation

"generates unique QR codes for each visitor, which are used for quick identification..."
  • Proof of Implementation: Validated. Upon successful form submission, the system routes the user to registration_success.blade.php, where it leverages the qrcode.min.js library to instantly generate a scannable 2D barcode tied strictly to the visitor's unique database primary key ID.

4. Automated SMS Notifications

"automatically sends an SMS notification to the host, allowing them to approve or reject..."
  • Proof of Implementation: Integrated natively with the AfricasTalking SMS API. Upon creation of a new database entry, an asynchronous HTTP POST request is triggered in the backend controller, dispatching a localized text message to the host containing dynamic approval/rejection links.

5. Timestamps & Audit Trail

"records the entry and exit time of each visitor, thereby maintaining a complete audit trail."
  • Proof of Implementation: The database table was engineered with dedicated checked_in_at and checked_out_at timestamps. Furthermore, an independent, immutable AuditLog table tracks every security action (Check-Ins, Syncs, Approvals), associating the exact User and IP Address to the log.

6. Visitor Categorization

"supports visitor categorization, allowing differentiation between guests and service providers..."
  • Proof of Implementation: The Livewire registration component actively listens to the type category field. If a visitor is categorized as a "Minor" or specific external contractor, the Document Object Model (DOM) dynamically adjusts to surface mandatory secondary fields (e.g., Guardian Name), enforcing categorization rules prior to submission.

7. Real-Time Monitoring & Reporting

"provides real-time monitoring of visitors within the premises and generates detailed reports..."
  • Proof of Implementation: The Guard and Admin Dashboards utilize asynchronous background updates to feed real-time analytics graphs. The System Security Audit Log views can be date-filtered and exported directly to PDF for management review.

8. Data Protection Act (DPA) Retention

"incorporates mechanisms for automatic data retention and deletion."
  • Proof of Implementation: System configured for legal compliance by extending Laravel's Prunable trait onto the Visitor Eloquent model. The system executes a cron job utilizing the PHP artisan kernel at 00:00 (midnight) daily, automatically hard-deleting all visitor logs older than 90 days without human intervention.

9. Offline Data Capture Resilience

"supports offline data capture, ensuring continuity of operations even when internet connectivity is unavailable."
  • Proof of Implementation: Achieved. The registration interface is built as a complete Progressive Web App (PWA). It registers a local Service Worker (sw.js) which intercepts failed internet submissions, stores the complete encrypted registration object into the browser's persistent IndexedDB, and auto-dispatches a Background Sync payload to the server upon network restoration.
